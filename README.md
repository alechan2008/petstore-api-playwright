# Petstore API Automation Framework

A Playwright + TypeScript API automation project built as a response to **Abstracta's practical challenge**, using the public Swagger Petstore service as the system under test.

This repository provides a clean and maintainable foundation for API automation using Playwright Test as the native runner, TypeScript for readability and type safety, and a layered structure based on API objects, builders, and reusable utilities.

The suite is intentionally divided into **endpoint tests** and **end-to-end tests**:

- **Endpoint tests** validate isolated API operations.
- **E2E tests** validate complete business flows in a more realistic way.

This split is deliberate and important for this project because the public Petstore API behaves like a shared sample service and may expose inconsistent or eventually consistent behavior across endpoints.

## Overview

This project was designed as a practical API automation framework starter with the following goals:

- keep the architecture simple and readable
- favor Playwright native capabilities before adding custom abstractions
- provide clear test intent through API objects
- keep test data deterministic and easy to reason about
- produce useful Playwright HTML reports with JSON attachments
- separate isolated endpoint validation from broader end-to-end flows

The system under test is the public **Swagger Petstore** API.

---

## Challenge Context

This repository was developed as a response to the **Automated API Testing practical challenge proposed by Abstracta**.

The exercise is based on the public Swagger Petstore API and asks for the validation of the following scenarios:

### Part 1
- Create 10 pets:
  - 5 with status `available`
  - 4 with status `pending`
  - 1 with status `sold`
- Retrieve the details of the pet with status `sold`

### Part 2
- List available pets and keep 5 of them in a data structure
- Create one order for each of the 5 selected pets

The challenge also explicitly values:

- exploration of the API documentation
- framework construction
- good request validations
- good programming practices
- a repository that is complete and easy to review

This project was designed with those expectations in mind.

## Covered Scenarios

### Endpoint Tests

These tests validate isolated API behavior one operation at a time.

#### `GET /pet/{id}`
Validates that a known pet can be retrieved by id and that the response contains the expected basic structure.

**Covered checks include:**
- response status is `200`
- returned pet id matches the requested id
- name is present and non-empty
- `photoUrls` is an array
- response body includes the expected key fields

---

#### `POST /pet`
Creates a pet using a controlled id and a predictable payload generated by the builder.

**Covered checks include:**
- response status is `200`
- returned pet matches the sent payload
- `id`, `name`, and `status` are correct
- `category`, `photoUrls`, and `tags` match exactly

---

#### `GET /pet/findByStatus?status=available`
Lists pets using the `available` status from the shared public dataset.

**Covered checks include:**
- response status is `200`
- response is an array
- at least one result is returned
- a sample subset contains valid `available` pets
- sampled records have a numeric id, non-empty name, and array-based collections such as `photoUrls`

---

#### `POST /store/order`
Creates an order for a known valid pet id.

**Covered checks include:**
- response status is `200`
- returned order matches the sent payload
- `id`, `petId`, `quantity`, `status`, and `complete` are correct
- `shipDate` is present and represents the same instant as the payload, even if the API serializes timezone differently

---

#### `POST /pet` repeated 10 times
Creates a controlled dataset of 10 pets in order to validate creation behavior at small scale.

**Covered checks include:**
- 10 pets are created successfully
- all generated ids are unique
- the expected business distribution is respected:
  - 5 pets with status `available`
  - 4 pets with status `pending`
  - 1 pet with status `sold`

---

### End-to-End Tests

These tests validate broader business flows in natural language terms.

#### E2E Part 1 ‚Äî Create pets and retrieve the `sold` pet
This scenario creates a deterministic dataset of 10 pets with a fixed distribution of statuses, then identifies the pet created with status `sold`, and finally retrieves that same pet by id.

**What this flow proves:**
- the API accepts controlled pet creation requests
- the expected distribution of statuses is achieved
- the created `sold` pet can be retrieved by id
- the retrieved `sold` pet preserves the same data that was originally sent

This test includes retry-based read logic for the final `GET /pet/{id}` because the public sample API may not expose immediate consistency after creation.

---

#### E2E Part 2 ‚Äî List available pets and create orders
This scenario retrieves general `available` pets from the public dataset, keeps the first 5 valid results in memory, and creates one order for each selected pet.

**What this flow proves:**
- the API can list generally available pets
- the test can work with real shared dataset responses
- five usable pets can be selected safely
- one valid order can be created for each selected pet

This flow is intentionally independent from Part 1.

## Design Decisions

### 1. Playwright Test as the native runner
The framework uses Playwright Test directly instead of wrapping it with extra tools. This keeps the setup modern, aligned with the official Playwright approach, and easy to maintain.

### 2. TypeScript for readability and type safety
TypeScript models and builders make the suite easier to understand and reduce mistakes when building payloads or validating response content.

### 3. API Object pattern instead of raw requests in tests
The test files do not contain raw endpoint construction everywhere. Instead, endpoint interactions are centralized in:
- `PetApi`
- `StoreApi`

This keeps the tests more readable and focused on behavior.

### 4. Builders for predictable payload creation
Payload builders are used to generate consistent and readable request bodies:
- `PetBuilder`
- `OrderBuilder`

This reduces inline JSON duplication and makes assertions easier to maintain.

### 5. Controlled id generation
`IdGenerator` uses a run-based seed and deterministic offsets to generate ids that are unique within a test execution while still being easy to reason about.

### 6. Separation of endpoint and E2E suites
The repository keeps endpoint tests and E2E tests separate on purpose:
- endpoint tests validate isolated operations
- E2E tests validate full scenarios

This makes the suite easier to understand and avoids overloading a single test style with too many responsibilities.

### 7. Reporting through JSON attachments
Important execution artifacts are attached to the Playwright report instead of being printed to the console. This makes the report more useful for review and debugging.

## Test Data Strategy

The suite uses a simple and controlled data strategy.

### Pet data
Pet payloads are built with stable defaults:
- fixed `category`
- fixed `photoUrls`
- fixed `tags`
- explicit `status`
- explicit or generated `name`

This keeps payloads predictable and makes response comparison straightforward.

### Order data
Order payloads are also deterministic:
- generated `id`
- known `petId`
- default `quantity = 1`
- default `status = placed`
- default `complete = true`
- generated `shipDate`

### Run-based ids
Each test execution creates a `runId` through `IdGenerator`.
That value acts as the base for deterministic ids inside the same run.

Example idea:
- run id = seed
- offset `+1`, `+2`, `+3`, etc. for pets
- different offsets for orders

This gives the suite controlled uniqueness without requiring external state.

## Reporting

The suite uses the Playwright HTML report and stores structured JSON attachments to make execution easier to inspect.

Typical attachments include:
- run context
- payloads sent to the API
- created resources
- selected sample data
- summary objects such as status distributions

This approach was intentionally preferred over relying on console logging.

To open the latest report:

```bash
npx playwright show-report
```

## Known Limitations

This project uses the public Swagger Petstore sample service.

Because of that, a few practical constraints apply:

### Shared public dataset
Some endpoints read from a shared public dataset, so responses are not isolated to this repository or this test run.

### Eventual consistency behavior
The public service may not reflect newly created pets immediately across all endpoints.  
For that reason, the E2E Part 1 flow includes retry-based retrieval logic for the `sold` pet.

### Time serialization differences
The `shipDate` returned by the order endpoint may be serialized with a different timezone format than the original payload, even when both values represent the same instant.  
The suite therefore compares normalized timestamps instead of raw strings.
